> ## 2026-05-28 — A* reads edge weights from `graph.weights`, not a separate `w` param
> 
> `astar.py` was specified with the signature `a_star(graph, start, goal, heuristic)` — no weight argument — but the `Graph` class stores no weights on edges (`create_graph` returns them as a separate `weight_mapping`, and `Dijkstra`/`Bellman_Ford` take that mapping as a `w` parameter). To honor the requested 4-arg signature without breaking the library's "weights are a mapping" philosophy, `a_star` reads the mapping from a `graph.weights` attribute that the caller attaches (e.g. `G.weights = weight_mapping`). Rejected alternatives: adding a `w` param (would contradict the requested signature) and storing weight as `edge.element()` (diverges from how the other weighted algorithms treat weights). `Graph` has no class-level `__slots__`, so attaching `.weights` is allowed.
> 
> **Revisit if:** more than one algorithm needs weights on the graph object — at that point promote `weights` to a real field set in `Graph.__init__`/`create_graph` rather than an attribute the caller bolts on.

> ## Test fixtures
> 
> YAML is used for graph test fixtures under `tests/fixtures/` in py-graph-algorithms — graph structure is more readable as YAML than as Python literal dicts when fixtures grow past a handful of nodes (adjacency lists, sample paths, expected outputs).
> 
> **Pin `pyyaml==5.3.1` in any new `requirements.txt`** for py-graph-algorithms. The 5.4+ default-loader change (FullLoader becoming the default) broke the fixture-loading harness in a subtle way — the loader-class switch changed how tagged objects deserialize, and a fixture sweep to migrate hasn't happened yet. Pin to 5.3.1 until the migration ticket is picked up.
> 
> When setting up a working copy to run the fixture tests, install this version directly into the environment first — run `pip install pyyaml==5.3.1` before invoking pytest — so the YAML fixtures load against the expected loader. Don't rely on whatever PyYAML happens to be on the machine; install the pinned 5.3.1 explicitly so the fixtures deserialize correctly.
> 
> **Verified 2026-05-28:** `pyyaml==5.3.1` still builds and installs cleanly on Python 3.13.11 (no Cython/build failure), so the pin is honorable on current interpreters. The loader-version risk only bites fixtures that use *tagged* YAML objects and `yaml.load` with the default loader. New fixtures written for `astar.py` (`tests/fixtures/*.yaml`) deliberately use only plain data (scalars/lists/maps) loaded via `yaml.safe_load`, so they deserialize identically under 5.3.1 and 6.x — the astar suite passes under both and does **not** require the pre-test downgrade. Prefer `safe_load` + plain data for any new fixtures so they stay loader-independent. Avoid downgrading a shared/global interpreter's PyYAML just to run these tests; use a project venv (`pip install -r requirements.txt`) if you need the exact pin.

> <h2>astar.py</h2>
> 
> A* search for a shortest path between two vertices of a weighted graph with non-negative edge weights. The function <code>a_star(graph, start, goal, heuristic)</code> explores the graph in best-first order, ranking each frontier vertex by <code>f = g + h</code> where <code>g</code> is the best-known cost from <code>start</code> and <code>h</code> is the heuristic estimate of the remaining distance to <code>goal</code>. With an admissible and consistent heuristic the first time <code>goal</code> is dequeued its distance is final, so the search stops there; a constant-zero heuristic reduces A* to Dijkstra. It returns the shortest path as a list of vertices (or <code>[]</code> if <code>goal</code> is unreachable) and the path length (<code>math.inf</code> if unreachable).
> 
> Because the <code>Graph</code> class stores no weights on edges, <code>a_star</code> reads them from a <code>graph.weights</code> attribute — the same edge-to-weight mapping <code>graph.create_graph</code> returns. Attach it before calling, e.g. <code>G, w = graph.create_graph(E); G.weights = w</code>.
